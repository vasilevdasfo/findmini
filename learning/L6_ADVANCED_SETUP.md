## 1️⃣ Cowork mode — Claude внутри Obsidian

### Что это
Плагин для Obsidian. Ты пишешь заметку, Claude видит её в реальном времени и может редактировать вместе с тобой. Идеально для vault'а знаний.

### Установка
1. Obsidian → Settings → Community plugins → Browse → **Cowork**
2. Enable → Open
3. Авторизуйся через Claude.ai (тот же аккаунт что в CLI)

### Что появляется
- Чат-панель справа — можно говорить с Claude про открытую заметку.
- Claude видит весь vault (читает любые заметки).
- Правки вставляются прямо в текст, видишь diff.

### Где используешь
- **Стратегические документы** — пишешь черновик, Claude предлагает структуру.
- **Обработка VOICE_INBOX** — продиктовал длинное голосовое → Claude раскладывает по проектам.
- **Daily review** — каждый вечер Claude собирает что сделано из разных заметок.

### Подводный камень
Cowork каждая сессия ИЗОЛИРОВАНА — не видит другие окна Obsidian. Если правишь vault из CLI параллельно — файлы синхронизируются через диск, но Claude не видит что делает «другой Claude» в соседнем окне. Дима решает это через **session routing** (см. пункт 4).

## 2️⃣ Parallel agents — работа в 3-5 потоков

### Что это
Один главный Claude запускает 3-5 sub-агентов, каждый делает свою часть задачи. Результаты собираются обратно.

### Пример
Ты: «прогони клиентский лендинг через 3 аудита»

Claude:
```
Task A (subagent): conversion-audit — 15 пунктов конверсии
Task B (subagent): client-deliverable-audit — грамматика + факты + ссылки
Task C (subagent): devils-advocate — критический взгляд, 5 неудобных вопросов
```

Все три работают параллельно (5 минут вместо 15), потом главный Claude собирает сводку.

### Как использовать
В Claude Code просто скажи:
> Запусти **параллельно** 3 subagents через Task tool: (1) ... (2) ... (3) ...

Либо используй `dispatching-parallel-agents` skill из Superpowers — он сам решает когда параллелить.

### Когда параллелить
- ✅ Независимые задачи (аудит 3 сайтов, ресёрч 5 конкурентов)
- ✅ Один файл прогнать через разные проверки
- ❌ Зависимые задачи (сначала А, потом Б на основе А)
- ❌ Одна короткая задача — overhead съест выигрыш

## 3️⃣ Scheduled tasks — cron для Claude

### Что можно
- **8:00 PT** — прочитать Gmail, выбрать важное, отправить в Telegram
- **Пн 9:00** — конкурентная разведка через Perplexity → summary в Obsidian
- **Каждые 2 часа** — проверить новые сообщения в группах → если кто-то задал вопрос, показать тебе
- **Каждый вечер** — собрать git-коммиты за день → дневной отчёт

### Как настроить
В Claude Code:
```
/plugin install scheduled-tasks
```
Потом:
```
Создай scheduled task:
- имя: morning-brief
- расписание: каждый день 8:00 PT
- действие: прочитать Gmail (последние 24ч), выделить важное (от клиентов / с дедлайном / деньги),
  отправить в Telegram через бот demi_bot.
```

`~/Library/LaunchAgents/com.alex.morning-brief.plist`:

## 4️⃣ Memory system — Claude помнит тебя

### Автоматическая память
Claude Code ведёт `~/.claude/projects/[имя-проекта]/memory/` — там накапливаются правила которые ты дал.

Пример: ты сказал «никогда не предлагай созвон в сообщениях клиентам» → Claude запишет в memory → в следующих сессиях помнит.

### Ручная память
Пиши в тексте:
> **Запомни правило:** при работе с клиентом X — всегда отвечать в формате Y.

Claude создаст файл в memory, при следующем старте проекта прочитает.

### Как посмотреть
```bash
ls ~/.claude/projects/
# или
cat ~/.claude/projects/[твой-проект]/memory/MEMORY.md
```

Там индекс всех правил.

### Важно
- Память у каждого проекта своя. Правило из проекта А не видно в проекте Б.
- Для сквозных правил — положи в `~/CLAUDE.md` (глобальный).

## 6️⃣ RLM-first подход — для больших файлов

### Проблема
Один большой файл (vault с 100 заметок, транскрипт 2 часа, лог Telegram за год) = 100K+ токенов. Если Claude прочитает целиком → медленно + дорого + теряет детали.

### Решение — Recursive Language Models (Alex Zhang, MIT 2025)
Не читать целиком. Работать рекурсивно:

```
PEEK    — wc -l + grep -n по ключевым словам (узнать где что лежит)
   ↓
GREP    — читать только нужные строки (Read с offset/limit)
   ↓
PARTITION — если всё равно много, дробить и запускать subagents
   ↓
MAP     — каждый subagent делает свой кусок
   ↓
REDUCE  — главный Claude собирает ответ
```

### Как пользоваться
Когда говоришь Claude «разбери vault за март»:
> Используй RLM подход: сначала grep по ключевым словам, потом читай только релевантные файлы частями. Не грузи весь vault в контекст.

Есть готовый skill `superpowers:rlm-context` (или `rlm-first`) — применяй его для:
- Больших vault'ов
- Telegram-архивов
- Длинных транскриптов встреч
- Лог git за месяц+

## 🔄 Session routing — связь между Cowork-окнами

### Проблема
Ты открыл в Obsidian 3 Cowork-сессии: одну по проекту А, вторую по проекту Б, третью по стратегии. Они друг друга НЕ видят. Нужно что-то передать из А в Б?

### Решение (паттерн Димы)
В `~/vault/3_ЗАДАЧИ/routing/` — файлы `TO_<session>.md`:

```
routing/
├── TO_PROJECT_A.md    ← сюда я кладу задачи для сессии А
├── TO_PROJECT_B.md
└── TO_STRATEGY.md
```

При старте каждой Cowork-сессии — в её CLAUDE.md пропиши:
```
## При старте — всегда читай
`3_ЗАДАЧИ/routing/TO_[текущий проект].md`
И обрабатывай всё новое.
```

Теперь в сессии А пишешь: «Перекинь это в Б» → Claude дописывает в `TO_PROJECT_B.md`. Через минуту откроешь сессию Б → она прочитает файл и возьмёт задачу.

### Что это даёт
- **Прямая связь с агентом Димы** без Telegram, без серверов, P2P.
- Твой Claude общается с Диминым Claude напрямую — для технических вопросов, debug-логов, sync по задачам.

- Экспорт чата в Markdown / JSON одной кнопкой.
- Панель профиля собеседника — чей узел, когда last seen, trust-статус.

## 7️⃣ Чистка дубликатов через хэш — пример личной эффективности

### Откуда задача
У Димы 28.04.2026 в `Google Drive/lost_and_found/` накопилось 193 файла на 39 ГБ — видео `MOVIO059.mp4`, `MOVIO059 (1777227156656).mp4`, `MOVIO059 (1777343784792).mp4` и т.д. DriveFS падает туда копиями файлов которые не смог привязать к папке. Глазами разобрать невозможно — имена похожие, размер одинаковый, но это **не гарантирует** что файлы идентичны.

### Главный принцип
**Совпадение размера ≠ дубликат.** Два разных видео могут весить одинаково. Надёжно только сравнение хэша (md5/sha) — это «отпечаток пальца» содержимого.

Но 39 ГБ хэшировать в лоб = 10+ минут диск-ввода. Поэтому **двухступенчатый трюк**:

1. **Шаг 1 — группировка по размеру (мгновенно).** `stat` отдаёт размер из inode, не читая файл. Файлы у которых размер уникален — точно не дубликаты, их можно сразу пропустить.
2. **Шаг 2 — md5 только на «подозреваемых».** Хэшируем только те файлы где есть совпадение по размеру. Из 193 файлов остаётся ~57.

Это уменьшает работу в разы — типичная RLM-логика (PEEK → FILTER → DEEP), только применённая к диску, не к контексту.

### Команда (Mac, готова к копированию)
```bash
cd "/path/to/folder"

# Шаг 1 — собрать (размер, путь) и оставить только размеры с дубликатами
find . -type f \! -name ".DS_Store" -exec stat -f "%z %N" {} \; | sort -n > /tmp/sizes.txt
awk '{print $1}' /tmp/sizes.txt | sort | uniq -c | awk '$1>1 {print $2}' > /tmp/dup_sizes.txt
echo "групп подозреваемых: $(wc -l < /tmp/dup_sizes.txt)"

# Шаг 2 — md5 только по этим группам
> /tmp/md5.txt
while read sz; do
  awk -v s="$sz" '$1==s {sub($1 FS,""); print}' /tmp/sizes.txt | while IFS= read -r f; do
    md5 -r "$f" >> /tmp/md5.txt
  done
done < /tmp/dup_sizes.txt

# Шаг 3 — отчёт: группы с одинаковым хэшем = настоящие дубликаты
sort /tmp/md5.txt | awk '
  { if ($1==prev_hash) { if (!shown) print prev_file; print $0; shown=1 }
    else { if (shown) print "---"; shown=0 }
    prev_hash=$1; prev_file=$0 }
'
```

### Правило безопасности
Сначала **отчёт**, потом **удаление**. Никогда не запускать `rm` на основании только размера или только имени — обязательно отчёт глазами + хэш.

При удалении оставлять файл с «чистым» именем (`MOVIO059.mp4`), удалять копии с суффиксом-id (`MOVIO059 (1777...).mp4`).

### Где это применять
- Google Drive `lost_and_found/` (задача из примера)
- Папка с фотками после миграции с iPhone (часто дубли с разными именами)
- Бэкапы клиентских проектов (несколько `final_v2_FINAL.docx`)
- Загрузки с Telegram — Telegram любит делать копии при пересылках

### Чему это учит
Не «загружать всё в окно Claude и спрашивать». Сначала локальный дешёвый фильтр (size → hash), потом узкий вопрос. Это **личная эффективность через композицию shell-команд** — то же самое что Claude делает с RLM-context на больших файлах.
